# CLAUDE.md

This file provides guidance to Claude Code (claude.ai/code) when working with code in this repository.

## Project Overview

POL (PenUltima Online) is a C++20 Ultima Online-compatible MMO server with a custom scripting language called eScript. It includes a full compiler/runtime for eScript, network handling, and game server logic.

## Build System

CMake, multi-config on Windows (Visual Studio), single-config on Linux. Note: `bin-build/` is a checked-in folder of helper build scripts, not a build directory. Binaries land in `bin/Release/` (Windows) or `bin/` (Linux).

**Windows:**
```powershell
cmake --build build --config Release --target pol -- /m
```

**Linux:**
```bash
cd build && cmake .. && make -j$(nproc)
```

After pulling changes, rebuild `ecompile`/`poltool`/`uoconvert` too, not just `pol` — the shard tests compile test scripts with `bin/Release/ecompile.exe`, and a stale one fails on newer eScript syntax.

**Useful CMake options** (single-config generators use `-DCMAKE_BUILD_TYPE=`; VS uses `--config`; full list in the root `CMakeLists.txt`):
- `-DENABLE_ASAN/ENABLE_TSAN/ENABLE_USAN/ENABLE_MSAN=ON` — sanitizers
- `-DUSE_CCACHE=ON` — faster rebuilds
- `-DENABLE_TIDY=ON` — clang-tidy
- `-DNO_PCH=ON` — disable pre-compiled headers (on by default)

## Running Tests

```powershell
# all / filtered (from repo root)
ctest --test-dir build -C Release [-j8]
ctest --test-dir build -C Release -R "unittest_pol|shard_test" --output-on-failure
ctest --test-dir build -C Release -R "escript_<testname>" -VV
```

- eScript tests: `testsuite/escript/*/` (one `.src` + expected-output file per test); server/shard tests: `testsuite/pol/`. Test definitions and fixtures live in `cmake/core_tests.cmake` and `cmake/escript_tests.cmake`; `ctest --test-dir build -N` lists everything.
- Shard tests boot a real server in the generated `build/coretest` dir; `POLCORE_TEST_FILTER=<pkg>:<file>:<func>` narrows the run (each part matches by prefix, any part may be empty).
- **Manual test shard**: run `bin\Release\pol.exe` with cwd `build\coretest` and no `POLCORE_TEST_RUN` set — it skips the test suite and stays up, webserver on `http://127.0.0.1:5006` (local-only, basic auth `polcore` / `test` per pol.cfg `WebServerPassword`). ctest sets `POLCORE_TEST_RUN`, which runs the suite and shuts down.
- On Windows, a shell cd'd into `build\coretest` blocks the `cleantestdir` fixture (dir-handle lock).

## Linting and Formatting

`.clang-format` at the root; CI enforces it (`.github/workflows/clang-format.yml`). Auto-format: `cmake --build build --target clang_format`. Coverage (Linux only): `-DGCOV=ON` + `coverage` target.

## Architecture

The source tree under `pol-core/` contains several libraries and executables:

| Component | Type | Purpose |
|-----------|------|---------|
| `clib/` | static lib | Core utilities (strings, files, logging, threading, sockets) |
| `bscript/` | static lib | eScript compiler (ANTLR4-based) and bytecode VM |
| `plib/` | static lib | Ultima Online file format reader/writer |
| `pol/` | executable | Main game server |
| `ecompile/` | executable | Standalone eScript compiler |
| `runecl/` | executable | eScript console runner |
| `uoconvert/` | executable | Converts UO client data files |
| `poltool/` / `uotool/` | executables | Utility tools |

Server subsystems map to subdirectories of `pol-core/pol/` (e.g. `network/`, `mobile/`, `item/`, `module/`); the folder names say what they do.

### eScript Language Pipeline

eScript source (`.src`/`.inc`) → ANTLR4 parser (`bscript/`) → AST → bytecode → VM execution. The `ecompile` binary precompiles scripts; the POL server can also compile on load. Module definitions in `pol-core/support/scripts/*.em` define the API exposed to scripts.

## Common Workflows

### Adding an eScript module function

eScript functions are implemented in `pol-core/pol/module/` as C++ classes inheriting from `Bscript::ExecutorModule`. The `.em` file in `pol-core/support/scripts/` declares the signature for the compiler; the corresponding C++ method handles execution. Search for an existing function (e.g., `ApplyDamage`) to see the pattern.

### Adding a shard test

Shard tests live in `testsuite/pol/testpkgs/<pkg>/` (a package with a `pkg.cfg`). Files named `test*.src` are discovered by `testsuite/pol/scripts/tests.src`; each `exported function test_*(...)` in them is called as a test case (receives a resource-manager helper, returns 1 on success or an error). Optional `setup.src`/`cleanup.src` run before/after the package. The test env under `build/coretest` is regenerated by ctest fixtures (see `cmake/core_tests.cmake`) — just rerun `ctest -R shard_test`.

## Conventions

- User-visible changes get entries in **both** `pol-core/doc/core-changes.txt` and `docs/docs.polserver.com/pol100/corechanges.xml` (same format as existing entries).

## Key External Dependencies

Vendored/fetched into `lib/` via CMake: Boost (headers + stacktrace), ANTLR4, libcurl, zlib, fmt, MariaDB Connector/C (bundled SQL backend), efsw, cppdap. Don't pin versions from memory — check `cmake/*.cmake`.
